## Background

The local NemoClaw fork (limcheewooi/NemoClaw) tracks upstream NVIDIA/NemoClaw but operates a custom 4-branch promotion model (dev → test → live → main). Until now the workflow existed only in ad-hoc user memory — no VERSION source of truth, no release-note template, no enforced gate. Commits were free to land on any branch.

## Reason

The user defined a mandatory 5-step deploy gate with hard requirements: SemVer in a `VERSION` file, per-stage tags, comprehensive release notes following a fixed filename pattern, and hook-enforced checks. This release dogfoods the workflow — the scaffold that defines the process ships as v0.0.1 itself, becoming the first end-to-end test of the gate.

## Changes

- `VERSION` (new) — single source of truth for the current release version. Contents: `0.0.1`.
- `docs/release-notes/` (new directory) — home for all release notes.
- `docs/release-notes/TEMPLATE.md` (new) — mandatory skeleton. All future release notes must include Background, Reason, Changes, Smoke test results, Regression test results, Rollback.
- `docs/release-notes/202604211127-Release-v0.0.1-scaffold-deploy-workflow.md` (new) — this file.
- `CLAUDE.md` — added "Deploy Workflow (MANDATORY)" section documenting: the 4-branch flow, 5-step gate, SemVer rules, per-stage tag strategy, release-note filename pattern `YYYYMMDDHHMM-Release-v<VERSION>-<slug>.md`, docs-only exemption, Step 1 iteration rule.
- `.git/hooks/pre-commit` — extended with two new blocking checks in addition to the existing `.commit-approved` marker:
  1. If any staged file lies outside `docs/release-notes/`, `VERSION` must also be staged.
  2. A file matching `docs/release-notes/*-Release-v<VERSION>-*.md` must exist on disk.
- Memory: `feedback_release_workflow.md` updated to reflect the new 4-branch + per-stage-tag + enforced-template spec.

No runtime code (`src/`, `nemoclaw/`, `nemoclaw-blueprint/`, `bin/`) is modified in this release.

## Rollback

```bash
# 1. Delete stage tags (local + origin). Some may not yet exist depending on how far the release progressed.
for t in v0.0.1 v0.0.1-live v0.0.1-test v0.0.1-dev; do
  git tag -d "$t" 2>/dev/null || true
  git push origin ":refs/tags/$t" 2>/dev/null || true
done

# 2. Reset branches to pre-release commit 67db244d
for b in dev test live main; do
  git checkout "$b"
  git reset --hard 67db244d
  git push --force-with-lease origin "$b"
done

# 3. Remove scaffold files if still on disk
rm -f VERSION
rm -rf docs/release-notes/

# 4. Restore pre-commit hook to the original 32-line commit-gate-only version
git show 67db244d:.git/hooks/pre-commit 2>/dev/null || true
# (The original hook is not tracked in git. Recreate by hand from `.git/hooks/pre-commit` memory
#  in feedback_commit_gate.md — the 32-line script that only checks .commit-approved.)
chmod +x .git/hooks/pre-commit

# 5. Restore CLAUDE.md
git checkout 67db244d -- CLAUDE.md
```

**Out-of-band cleanup:** none — this release is scaffold-only and has no deployed artifact, cache, or external state.

**Coordination note:** `--force-with-lease` on shared branches requires confirming no one else has pushed since. For this solo fork at v0.0.1 time, that is trivially safe.

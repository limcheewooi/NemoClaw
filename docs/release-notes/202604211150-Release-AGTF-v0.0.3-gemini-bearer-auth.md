## Background

During Phase 0 sandbox bring-up (prior session), the user wired NemoClaw to Google Gemini 2.5 Flash via its OpenAI-compatible endpoint (`https://generativelanguage.googleapis.com/v1beta/openai/`). Upstream NVIDIA/NemoClaw had recently merged PR #1960 to fix a different Gemini auth issue: Gemini was rejecting requests that carried BOTH an `Authorization: Bearer` header AND a `?key=` query param, returning *"Multiple authentication credentials received"*. Upstream's fix was to switch `gemini-api` to query-param auth (`?key=`) and drop the Bearer header.

The user's setup broke under that fix: their OpenAI-compat endpoint rejected the query-param request with HTTP 400 *"Missing or invalid Authorization header"*, the opposite of what upstream was guarding against. The two endpoints behave differently: `/v1beta/models/*:generateContent` (upstream's target) accepts query-param auth, but `/v1beta/openai/*` (used for OpenAI SDK compatibility) requires Bearer. A patch was developed in-conversation, validated live against Gemini 2.5 Flash, and left sitting in the working tree as an uncommitted change for two days while the deploy workflow was scaffolded (v0.0.1, AGTF-v0.0.2).

This release commits that patch through the new deploy gate — the first runtime-code release to exercise the workflow end-to-end.

## Reason

The patch is necessary for the user's active Gemini configuration: without it, `nemoclaw onboard` and subsequent sandbox probe requests fail with HTTP 400 when targeting the OpenAI-compat Gemini endpoint. The fix restores onboarding to a working state for this fork's configuration while leaving behavior unchanged for every other provider (NVIDIA, compatible-endpoint, local inference, etc.).

## Changes

- [src/lib/onboard.ts](../../src/lib/onboard.ts) (lines 1337–1343) — `getProbeAuthMode(provider)` now returns `undefined` for all providers (previously returned `"query-param"` specifically for `gemini-api`). Parameter renamed to `_provider` to satisfy the project's unused-var convention. Comment rewritten from the stale "LOCAL PATCH (uncommitted)" placeholder to an accurate "Fork override of upstream #1960" explanation, preserving the WHY (Gemini OpenAI-compat endpoint requires Bearer).
- [test/gemini-probe-auth.test.ts](../../test/gemini-probe-auth.test.ts) — test file updated to encode the new fork behavior. Previously asserted `getProbeAuthMode("gemini-api") === "query-param"` and that the compiled source paired `gemini-api` with `query-param`; both were encoding upstream #1960's behavior and would fail under the fork override. Now asserts `getProbeAuthMode` returns `undefined` for every provider and that the compiled source does **not** pair `gemini-api` with `query-param`. `describe` blocks renamed to reflect the fork-override nature.
- [VERSION](../../VERSION) — bumped `0.0.2` → `0.0.3`
- This release note (new).

**Behavior-change surface:**

| Provider | Before (upstream + #1960) | After (this patch) |
|---|---|---|
| `gemini-api` | `"query-param"` | `undefined` (Bearer via default flow) |
| `nvidia-prod` | `undefined` | `undefined` (unchanged) |
| `compatible-endpoint` | `undefined` | `undefined` (unchanged) |
| (any other) | `undefined` | `undefined` (unchanged) |

**Net effect:** only `gemini-api` onboarding requests change — from `?key=<API_KEY>` to `Authorization: Bearer <API_KEY>`.

**No public API surface changed:** function signature and name are unchanged; only the return value for `gemini-api` input differs.

## Known regression risk

This change **diverges from upstream** and is correct for this fork's configuration only. If the fork ever re-merges from `upstream/main`, the upstream `getProbeAuthMode` will try to come back. Two scenarios to watch:

1. **If a user of this fork points `gemini-api` at the non-OpenAI-compat Gemini endpoint** (`/v1beta/models/*:generateContent` directly, rather than `/v1beta/openai/`), they will hit the original issue #1960 failure mode ("Multiple authentication credentials received"). Mitigation: use `compatible-endpoint` provider with an explicit base URL, or revert this patch locally.
2. **Upstream merge conflict** on future `git merge upstream/main` is likely at this hunk. Resolve by keeping the fork version and re-confirming the OpenAI-compat endpoint is still the user's target.

Documented here so it is not forgotten.
